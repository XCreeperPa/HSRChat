# BWiki 图片资源去重与压缩处理需求

本文档分析 HSRChat BWiki 图片多模态信源的“图片资源去重与压缩处理”需求，并记录当前已经落地的全量同步与压缩流水线。本文档定义目标、边界、策略、实现状态与验收标准。

---

## 1. 背景

当前 BWiki 图片信源已经能够识别并下载高价值图片，包括任务 CG、书籍插图、短信图片和角色立绘。原图缓存与压缩缓存分别位于：

```text
references/bwiki_images/assets/
references/bwiki_images/assets_webp/
```

当前阶段结果：

* 高价值候选：224。
* 原图缓存文件数：208。
* 原图缓存体积：591,360,670 bytes，约 563.97 MiB。
* WebP 压缩缓存文件数：208。
* WebP 压缩缓存体积：51,288,736 bytes，约 48.91 MiB。
* 压缩节省：约 515.05 MiB，节省比例约 91.33%。
* 原图缓存不纳入普通 Git，由 `.gitignore` 排除；WebP 压缩参考图纳入普通 Git。
* 索引、估算报告与压缩对照索引纳入 Git：
  * `references/bwiki_images/index.json`
  * `references/bwiki_images/estimate_report.json`
  * `references/bwiki_images/compressed_index.json`

虽然原图体积低于 1 GiB 阈值，但随着 BWiki 页面更新、新角色增加、更多图片类型纳入，图片缓存会继续膨胀。当前已通过 WebP 派生图把面向 Agent 的参考图控制在约 50 MiB；后续仍需在不破坏图片信源可追溯性的前提下，继续维护去重、压缩、派生版本和存储分层策略。

---

## 2. 需求目标

### 2.1 一句话定义

在保留原始图片可追溯性和多模态可用性的前提下，对 BWiki 图片缓存进行内容级去重、可选无损/有损压缩与派生版本管理，降低本地存储占用和后续检索成本。

### 2.2 主要目标

* 减少重复图片文件占用，尤其是同图多名、格式别名、跨页面复用的情况。
* 为大尺寸 PNG/JPG 生成适合 Agent 浏览、OCR、caption 和 embedding 的轻量派生版本。
* 保留原图元数据、来源页面、远端 URL、sha1/sha256 和出现上下文。
* 避免压缩处理污染原始信源，确保原图、压缩图、缩略图各自有清晰身份。
* 为未来是否使用 Git LFS、对象存储或外部缓存提供数据依据。

### 2.3 非目标

* 不把所有图片强制压缩后替换原图。
* 不为了体积牺牲图片中的文字、符号、细节和透明背景。
* 不处理战斗机制、材料、奖励、UI 图标等当前已排除的低价值图片。
* 不在本阶段引入复杂的视觉相似检索或模型 embedding 实现。
* 不改变当前普通 Git 不提交 `assets/` 的规则。

---

## 3. 当前问题

### 3.1 内容重复

当前脚本已基于 BWiki `imageinfo.sha1` 做了一层远端内容去重，但仍可能存在以下重复：

* 同一图片被 BWiki 以不同文件名、不同扩展名或不同页面引用。
* 远端 sha1 不同，但视觉内容几乎相同，例如重新编码、轻微裁切、透明边距不同。
* PNG 与 JPG 版本共存，肉眼内容相同但压缩方式不同。
* 同一角色立绘存在角色页、活动页、资料页多处引用。

这些重复不一定都应删除。对 HSRChat 来说，图片的“来源上下文”同样有价值，因此应去重二进制内容，但保留多处 appearance 记录。

### 3.2 原图过大

角色立绘和剧情 CG 往往是高分辨率 PNG，单文件可达数 MB。原图适合归档和精细考据，但不一定适合所有下游任务：

* Agent 展示时不总需要 2048px 原图。
* OCR 可能只需要适当缩放后的图片。
* caption 或 embedding 通常需要模型可接受的尺寸。
* 高频检索或预览更适合使用 WebP/JPEG 派生图。

### 3.3 格式复杂

BWiki 文件标题扩展名与实际 MIME 可能不完全一致，例如 `.jpg` 标题指向 PNG 内容，或者 `.png` 标题实际为 JPEG。压缩处理不能只依赖文件名，应以实际文件头和 MIME 为准。

### 3.4 可追溯性风险

如果直接覆盖原图或只保留压缩版本，会带来风险：

* 无法复核 BWiki 原始图片。
* 视觉细节被压缩损失后，后续考据可能误判。
* 索引中的 sha1/sha256 与本地文件不再对应。
* 回答用户时难以说明图片是原图还是派生图。

因此压缩产物必须作为派生资源存在，不能默认替代原始缓存。

## 4. 资源分层建议

建议将图片资源分成三层：

| 层级 | 路径建议 | 是否提交 Git | 作用 |
| :--- | :--- | :--- | :--- |
| 原始缓存 | `references/bwiki_images/assets/` | 否 | 保存 BWiki 原图，用于复核、归档和高精度视觉分析 |
| 派生缓存 | `references/bwiki_images/assets_webp/` | 是 | 保存面向 Agent 的轻量 WebP 参考图 |
| 文本索引 | `references/bwiki_images/index.json`、`references/bwiki_images/compressed_index.json` | 是 | 保存元数据、哈希、来源、处理状态和派生关系 |

原始缓存 `assets/` 应被 `.gitignore` 排除；派生缓存 `assets_webp/` 必须进入 Git。项目固定规则是：图片二进制只提交 WebP，不提交原图缓存。

## 5. 去重需求

### 5.1 精确去重

精确去重用于识别完全相同的二进制内容。

必要字段：

* `sha256`：本地文件内容哈希，作为本地精确身份。
* `bytes`：文件大小。
* `mime`：实际 MIME。
* `local_path`：当前本地路径。
* `appearances`：所有来源页面和上下文。

行为要求：

* 同一 `sha256` 的文件只保留一个物理原图。
* 其他文件名只在索引中作为别名或 appearance 记录。
* 不删除来源上下文。
* 若需要兼容旧路径，可在索引中保留 `legacy_local_paths`。

### 5.2 远端文件去重

远端去重用于识别 BWiki API 返回同一 `imageinfo.sha1` 的资源。

行为要求：

* `imageinfo.sha1` 相同的资源视为同一远端文件。
* 合并 `duplicate_file_names`。
* 保留所有 BWiki 文件标题、远端 URL 和页面来源。
* 若远端 sha1 与本地 sha256 关系变化，应在报告中标记。

### 5.3 近似去重

近似去重用于识别视觉内容基本一致但编码不同的图片。

可选方法：

* 感知哈希，例如 pHash、dHash、aHash。
* 统一缩放后计算结构相似度。
* 对透明背景图片先合成到固定背景再计算视觉哈希。

行为要求：

* 近似去重只输出候选报告，不自动删除。
* 候选必须包含相似度、尺寸差异、文件大小差异和来源页面。
* 角色立绘、剧情 CG 这类高价值图，即使相似，也应人工确认后再合并。

## 6. 压缩需求

### 6.1 压缩原则

* 原图默认保留，不覆盖。
* 压缩版本必须写入派生目录。
* 压缩版本必须有独立哈希和元数据。
* 对含文字、UI、符号、透明背景的图片优先使用无损或高质量策略。
* 对纯插画/立绘预览可以生成有损 WebP/JPEG 派生版本，但不得作为事实复核依据。

### 6.2 派生规格建议

当前已支持以下派生规格：

| 派生类型 | 建议格式 | 建议尺寸 | 用途 |
| :--- | :--- | :--- | :--- |
| `role_painting` | WebP | 长边 1600px，quality 86 | 角色立绘参考 |
| `large_story_cg` | WebP | 长边 1920px，quality 84 | 剧情 CG、影像、回想 |
| `text_or_small_context_image` | WebP | 原尺寸，quality 88 | 书籍、短信、线索、NPC 图 |
| `default_story_image` | WebP | 长边 1920px，quality 86 | 兜底剧情图 |

### 6.3 格式策略

PNG：

* 透明背景角色立绘应保留透明通道。
* 派生预览可选择 WebP lossless 或带 alpha 的 WebP。
* 不建议把透明立绘直接转成白底 JPEG，除非明确作为展示预览。

JPEG：

* 可生成质量 85 到 92 的 WebP/JPEG 预览。
* 需保留原 JPEG 作为复核文件。

WebP：

* 若原图已经是 WebP，可直接生成尺寸派生版本。
* 需要记录是否 lossless。

GIF：

* 当前图片信源以静态图为主。若未来出现 GIF，需决定保留动画还是抽首帧。

### 6.4 质量底线

压缩后必须满足：

* 肉眼无明显色块、边缘破损、透明通道错误。
* 图片中文字、符号、UI 标签仍可识别。
* 宽高比例不变。
* 不引入旋转、裁切、黑边或背景色错误。
* 派生图大小明显小于原图，否则不保留该派生版本。

## 7. 索引设计建议

建议在 `index.json` 中保留原图字段，并新增派生资源字段：

```json
{
  "id": "asset id",
  "file_name": "黄泉立绘.png",
  "local_path": "references/bwiki_images/assets/角色/黄泉立绘.png",
  "sha256": "original sha256",
  "bytes": 4626228,
  "derived": [
    {
      "kind": "preview",
      "local_path": "references/bwiki_images/assets_webp/角色/黄泉立绘.webp",
      "format": "webp",
      "width": 1024,
      "height": 1024,
      "bytes": 512000,
      "sha256": "derived sha256",
      "lossless": false,
      "quality": 90,
      "source_sha256": "original sha256",
      "generated_at": "YYYY-MM-DDTHH:MM:SSZ"
    }
  ]
}
```

当前单独生成：

```text
references/bwiki_images/compressed_index.json
```

这样能避免主索引过大，并使派生流程可以独立重跑。

## 8. 工作流建议

当前工作流已经落地为一键流水线：

```bash
python scripts/run_bwiki_image_pipeline.py --clean
```

该命令会清理原图和压缩缓存，重新执行图片发现、筛选、远端解析、并发下载和并发压缩。单张图片解析、下载或压缩失败时会记录状态并跳过，不中断全量流程。

历史设计分三步推进：

### 阶段 A：审计与报告

目标：不修改图片文件，只生成报告。

产出：

* 精确重复组。
* 远端 sha1 重复组。
* 近似重复候选组。
* 按文件大小排序的大图列表。
* 按类型统计的压缩收益预估。

### 阶段 B：派生图生成

目标：保留原图，生成预览/缩略图等派生图。

产出：

* `assets_webp/`
* `compressed_index.json`

### 阶段 C：物理去重

目标：只保留一份原始二进制，索引中合并重复路径和来源。

注意：

* 仅对精确重复自动处理。
* 近似重复只做候选报告，需人工确认。
* 处理前必须能回滚或重建。

## 9. 验收标准

### 9.1 去重验收

* 能列出所有 `sha256` 完全相同的重复文件组。
* 合并后每个原始文件仍能追溯到所有 BWiki 页面来源。
* 不丢失 `appearances`、`duplicate_file_names` 和远端 `imageinfo` 元数据。
* 精确去重前后索引可解析，且所有保留的 `local_path` 均存在。

### 9.2 压缩验收

* 派生图总大小低于原图对应集合的 40% 到 60%，具体按图片类型评估。当前实际为 48.91 MiB / 563.97 MiB，约 8.67%。
* 随机抽样查看不少于 20 张图片，无明显压缩伪影、透明背景错误或文字不可读问题。
* 每张派生图记录 `source_sha256`，可反查原图。
* 压缩失败不能影响原图索引。

### 9.3 Git 边界验收

* `assets/` 未进入 `git ls-files`。
* `assets_webp/` 已进入 `git ls-files`，并与 `compressed_index.json` 对齐。
* 可提交的包括文档、脚本、索引、报告和 WebP 参考图；不可提交原图缓存。
* `git status --ignored` 能明确显示二进制缓存被忽略。

---

## 10. 风险

* 过度压缩可能损害图片中的文字和细节。
* 近似去重误判可能把不同版本、不同场景或不同角色图误合并。
* 派生图如果没有清晰索引，容易被误当作原始信源。
* 引入第三方图片处理库会增加环境依赖。
* 如果未来改用 Git LFS 或对象存储，需要重新定义同步、部署和缓存策略。

---

## 11. 初步结论

图片资源去重与压缩处理是 BWiki 图片信源继续扩张前必须补上的工程能力。当前已经完成“原图缓存 + WebP 派生缓存 + 压缩对照索引”的基础能力；原图继续作为本地缓存保留，不直接被压缩版本替换。

最稳妥的后续路线是：

1. 定期运行 `python scripts/run_bwiki_image_pipeline.py --clean` 重建图片缓存和压缩索引。
2. 只对精确重复做自动合并。
3. 继续为展示、OCR 和模型输入生成独立派生图。
4. 保持 `assets/` 不进入普通 Git，并保持 `assets_webp/` 进入普通 Git。
