# BWiki 图片多模态信源需求分析与同步记录

本文档记录“将 BWiki 附带图片纳入 HSRChat 多模态信源”的需求、边界、同步策略与阶段性结果。当前仓库已进入高价值图片下载阶段：脚本、文本索引与 WebP 参考图可提交，原图缓存保持本地化，不纳入普通 Git。

---

## 1. 背景与需求动机

HSRChat 当前由两个文本型信源支撑：

| 信源 | 本地路径 | 数据形态 | 主要用途 |
| :--- | :--- | :--- | :--- |
| BWiki 文本 | `references/wiki/` | Wikitext `.txt` | 游戏内书籍、任务文本、角色介绍、考据文本 |
| B站官方视频元数据 | `references/bilibili/` | 视频详情 `.json` | 官方 PV、动画短片、EP 等发布时间与简介校验 |
| BWiki 图片索引 | `references/bwiki_images/index.json` | 图片元数据 `.json` | 任务 CG、书籍插图、短信图片、角色立绘等视觉证据 |

现有 BWiki 文本中已经保留了大量图片引用，但这些引用目前只作为 Wikitext 标记存在，并没有形成可被 Agent 直接读取、展示、检索或交叉引用的多模态资产库。

新增“BWiki 图片多模态信源”的核心价值是：把剧情 CG、任务截图、短信图片、书籍插图、收藏卡、角色相关视觉资料等从纯文本语料旁边“拉出来”，作为独立但可追溯的视觉证据，补足纯文字难以表达的画面信息。

典型收益包括：

* **剧情考据更完整**：例如任务 CG、照片、手稿、地图、符号图案可能承载文本未完整转写的信息。
* **角色与场景识别更准确**：角色外观、构图、服饰、道具、环境信息适合通过图片补充。
* **回答可引用视觉来源**：当用户询问“这张图里出现了什么”“某任务 CG 有哪些细节”时，可以直接定位到原始 BWiki 图片及其所属页面。
* **为后续多模态 RAG 铺路**：先建立图片资产与文本页面的关系，再逐步增加 OCR、caption、视觉 embedding 或多模态模型分析。

---

## 2. 需求定义

### 2.1 一句话定义

在不破坏现有文本语料结构的前提下，从 BWiki 页面中提取、同步并索引与剧情/世界观/角色设定相关的图片，将其作为 HSRChat 的第三类本地信源。

### 2.2 目标用户问题

该信源应支持或准备支持以下问题类型：

* “某个任务里的 CG 是哪张？能看看吗？”
* “这张 BWiki 图出自哪个任务/书籍页面？”
* “匹诺康尼某张收藏卡上有什么视觉元素？”
* “某个角色页面附带的立绘/Logo/视觉图和文本设定有什么关系？”
* “某条短信图片、照片、通缉令、拓片、护照贴纸是否能作为剧情证据？”

### 2.3 与现有信源的关系

该需求不是替换 `references/wiki/`，而是增强它：

* BWiki 文本仍然是主语义入口。
* 图片信源应以 BWiki 页面为父级来源，保留“图片出现在哪个分类、哪个页面、哪一行/哪段上下文”的关联。
* 当图片没有足够文本说明时，应避免让 Agent 单凭文件名做过度推断。
* 图片信源与 B站视频元数据可以互补，但初期不要求做跨平台视觉匹配。

## 3. 本地样本观察

对当前 `references/wiki/` 的 Wikitext 抽样检索显示，图片引用并非单一格式，而是至少包含以下形态：

| 图片入口类型 | 当前样例形态 | 抽样数量级 | 需求含义 |
| :--- | :--- | :--- | :--- |
| 放大图片模板 | `{{图片放大|开拓任务-...-CG.png|600px|1100px}}` | 约 78 处 | 多为任务 CG、物品图、关键截图，应优先纳入 |
| MediaWiki 文件链接 | `[[File:...png|600px|center]]`、`[[文件:...jpg|234px|center]]` | 约 57 处 | 书籍插图、任务图片、NPC 图片等，应优先纳入 |
| 短信/对话图片 | `{{角色对话|左|角色名|图片|短信-...png}}` | 约 20 处 | 剧情上下文强，适合保留对话邻近文本 |
| 角色立绘 | 由角色页模板渲染出 `角色名立绘.png`，后缀变体如 `角色名立绘2.png`、`角色名立绘3.jpg/png` 默认排除 | 约百个候选 | 本地 wikitext 不直接包含，需通过命名规则和 `imageinfo` 补全 |
| 角色 Logo/背景图 | `特殊:文件路径/WIKIlogo-角色名.png`、`patchwiki.biligame.com/images/sr/...png` | 少量 | 多为装饰或页面资源，默认不作为高价值图片 |
| 图标/小图 | `{{图标|...}}`、`[[file:...|24px]]`、`[[file:...|32px]]` | 约 1396 处 | 多为元素、奖励、材料、敌人或 UI 图标，应默认排除或低优先级 |

上述数量是基于当前仓库文本的粗略正则扫描结果，只用于判断需求复杂度，不应作为最终实现统计。实际同步结果以 `references/bwiki_images/estimate_report.json` 和 `references/bwiki_images/index.json` 为准。

重要结论：**不能把“页面中出现的所有图片”都视为同等价值的多模态信源。** HSRChat 的核心范围是世界观、剧情和角色设定，因此应优先收录叙事图片，谨慎处理战斗、奖励、材料、机制相关图标。

## 4. 范围边界

### 4.1 应纳入范围

优先纳入具备剧情、世界观或角色设定价值的图片：

* 主线、支线、同行任务中的剧情 CG。
* 游戏内书籍、报纸、拓片、卡片、照片、通缉令、手稿等插图。
* 短信剧情中的图片消息。
* 与角色故事、角色身份、阵营、外观设定直接相关的视觉资料，当前默认包含角色主立绘。
* BWiki 页面正文中用于说明剧情事实、地点、物件或人物关系的图片。

### 4.2 可选纳入范围

以下内容可以作为后续阶段补充，但不建议在 MVP 中强制纳入：

* 角色页面 Logo、头像、页面背景图。
* 页面顶部装饰图、导航图。
* 地点、NPC、物品的纯展示图。
* 没有上下文说明、只能依赖文件名猜测含义的图片。

### 4.3 默认排除范围

为延续 HSRChat 的“剧情与世界观优先、机制内容运行时过滤”原则，以下图片应默认排除：

* 元素、命途、材料、遗器、光锥、敌人弱点、奖励图标等机制向小图。
* 纯 UI 图标、按钮、导航、模板装饰图。
* 明确用于战斗攻略、角色强度、排行榜、配队说明的图片。
* BWiki 模板或站点公共资源中与《崩坏：星穹铁道》剧情无直接关系的图片。

## 5. 核心功能需求

### 5.1 图片发现

后续实现需要从当前同步得到的 Wikitext 中识别图片引用，至少覆盖：

* `{{图片放大|文件名|...}}`
* `[[File:文件名|...]]`
* `[[file:文件名|...]]`
* `[[文件:文件名|...]]`
* `{{角色对话|...|图片|文件名|...}}`
* `{{filepath:文件名}}`
* CSS 或模板中出现的 `特殊:文件路径/文件名`
* 直接出现的 `https://patchwiki.biligame.com/images/sr/...`
* 角色页模板渲染出的主立绘文件名候选，例如 `黄泉立绘.png`

识别阶段应同时记录图片所在的本地文本文件、分类、页面标题、出现位置和邻近上下文。

### 5.2 图片解析与下载

图片文件名不一定是直接可下载 URL。后续实现应考虑通过 MediaWiki 文件路径机制或 API 解析真实地址，而不是手工拼接不可验证链接。

理想能力包括：

* 根据 BWiki 文件名解析真实图片 URL。
* 获取图片 MIME 类型、大小、宽高、文件哈希、最后更新时间。
* 支持重定向或同名文件变更。
* 支持失败重试与限速，避免触发 BWiki/WAF 风控。
* 对已下载图片做增量判断，避免重复拉取。

当前脚本使用 MediaWiki `imageinfo` 解析原图地址、MIME、大小、宽高、远端 sha1 与更新时间。页面 HTML 中的缩略图 URL 仅用于确认渲染现象，不作为下载目标。

示例 API：

```text
https://wiki.biligame.com/sr/api.php?action=query&titles=File:黄泉立绘.png&prop=imageinfo&iiprop=url|mime|size|sha1|timestamp&format=json&utf8=1
```

### 5.3 元数据索引

图片不应只作为散落文件保存，应有结构化索引记录来源和上下文。

建议的图片元数据字段：

```json
{
  "id": "稳定图片 ID",
  "source": "bwiki_image",
  "file_name": "开拓任务-某任务-CG.png",
  "original_ref": "{{图片放大|开拓任务-某任务-CG.png|600px|1100px}}",
  "resolved_url": "https://...",
  "local_path": "references/bwiki_images/assets/...",
  "mime": "image/png",
  "width": 1440,
  "height": 810,
  "bytes": 1234567,
  "sha256": "...",
  "wiki_category": "开拓任务",
  "wiki_page_title": "某任务",
  "wiki_text_path": "references/wiki/开拓任务/某任务.txt",
  "line": 123,
  "context_before": "图片前的邻近文本",
  "context_after": "图片后的邻近文本",
  "asset_kind": "story_cg",
  "priority": "high",
  "excluded_reason": null,
  "synced_at": "YYYY-MM-DDTHH:MM:SSZ"
}
```

如果同一图片在多个页面中复用，应避免重复下载二进制文件，但允许在索引中记录多个 `appearances`。

### 5.4 多模态可用性

图片信源至少应支持两层使用方式：

* **基础层**：Agent 能根据文本检索结果找到相关图片，并引用其来源、文件名和本地路径。
* **增强层**：Agent 能读取图片内容，结合 OCR/caption/视觉模型结果回答画面细节问题。

MVP 不一定要立即生成视觉 embedding。先完成“可追溯图片资产库”更稳妥。

## 6. 建议数据结构

建议新增独立目录，不混入 `references/wiki/` 的文本目录，以免现有同步脚本的目录裁剪逻辑误删图片资产。

```text
references/
├── wiki/                       # 现有 BWiki 文本
├── bilibili/                   # 现有 B站视频元数据
└── bwiki_images/               # 建议新增：BWiki 图片多模态信源
    ├── index.json              # 图片索引，记录来源、上下文、分类、下载状态
    ├── estimate_report.json    # 估算报告，包含排除项和失败候选
    ├── assets/
    │   ├── 开拓任务/
    │   ├── 书籍/
    │   ├── 角色/
    │   └── ...
    └── derived/                # 后续可选：OCR、caption、embedding 元数据
        ├── ocr/
        └── captions/
```

命名建议：

* 目录名使用 `bwiki_images`，避免与现有 `wiki` 文本混淆。
* 二进制图片按分类分层保存，但文件去重应以哈希或解析后的 canonical URL 为准。
* 对 Windows 不支持的字符继续沿用现有 `clean_filename` 思路。
* `assets/` 是本地原图缓存目录，已在 `.gitignore` 中排除；提交时只纳入脚本、文档、索引、估算报告、压缩索引和 `assets_webp/` WebP 参考图。项目固定规则是：图片二进制只提交 WebP，不提交原图缓存。

## 7. 配置设计建议

未来可以在 `config.json` 中新增独立配置块，而不是扩展现有 `wiki.categories`：

```json
{
  "bwiki_images": {
    "enabled": false,
    "source_categories": [
      "开拓任务",
      "开拓续闻",
      "同行任务",
      "冒险任务",
      "书籍",
      "角色",
      "NPC"
    ],
    "include_patterns": [
      "图片放大",
      "File",
      "文件",
      "角色对话图片",
      "filepath"
    ],
    "exclude_small_icons": true,
    "min_width": 128,
    "min_height": 128,
    "download_assets": true,
    "generate_derivatives": false
  }
}
```

其中 `enabled` 初始建议为 `false`，表示即使文档和脚本存在，也不会在常规同步中意外下载大量图片。

## 8. 检索与引用策略

### 8.1 检索入口

图片信源不应孤立检索，推荐入口顺序为：

1. 先通过 `references/wiki/` 文本全文检索定位相关页面。
2. 再通过图片索引查找该页面关联图片。
3. 若用户明确上传/指定图片，再反向通过文件名、哈希、OCR/caption 或视觉特征检索来源页面。

### 8.2 回答引用规范

纯聊天模式下引用图片时，Agent 应说明：

* 图片来自哪个 BWiki 页面。
* 图片在页面中承担的上下文作用，例如“任务 CG”“短信图片”“书籍插图”。
* 若经过视觉模型解读，应区分“图片中可见内容”和“根据上下文推断”。

示例表达：

> 在 BWiki `开拓任务/某任务` 页面中，这张图片以 `{{图片放大|...}}` 的形式出现在某段剧情后。图片本身可见……；结合前后文本，较合理的解释是……

### 8.3 避免误用

应避免以下行为：

* 仅凭图片文件名推断剧情事实。
* 把游戏机制图标当作剧情证据。
* 把 BWiki 社区上传图等同于游戏内官方原始素材。
* 未说明来源和上下文就直接给出视觉结论。

## 9. 同步与增量策略分析

该信源存在两个不同层面的“变化”：

* **页面变化**：Wikitext 变更导致图片引用新增、删除或上下文改变。
* **文件变化**：BWiki 同名图片被替换、压缩、迁移或重定向。

因此，后续实现可以考虑双状态设计：

| 状态对象 | 判断依据 | 作用 |
| :--- | :--- | :--- |
| 页面图片引用状态 | 页面 revision timestamp + 引用列表哈希 | 判断是否需要重新解析页面图片 |
| 图片文件状态 | 图片 canonical URL / 文件 timestamp / sha256 | 判断是否需要重新下载图片二进制 |

这比只记录“页面已同步”更可靠，因为图片文件可能在页面文本未明显变化时被替换。

## 10. 风险与约束

### 10.1 数据体积风险

图片资产会显著增加仓库体积，尤其是剧情 CG、高清插图和重复资源。若直接纳入 Git，仓库克隆成本会上升。

可选缓解方式：

* MVP 只同步索引，不下载图片二进制。
* 只同步高优先级叙事图片，排除图标和装饰图。
* 对图片使用 Git LFS 或外部对象存储，但这会增加部署复杂度。
* 对 assets 设置体积预算，例如首阶段控制在数百 MB 内。

当前执行结果：高价值已知图片估算总量为 591,360,670 bytes，约 563.97 MiB，低于 1 GiB 阈值。图片原图已下载到本地 `references/bwiki_images/assets/`，并额外生成面向 Agent 的 WebP 参考图到 `references/bwiki_images/assets_webp/`；只有 WebP 参考图进入普通 Git，原图缓存不提交。

### 10.2 版权与授权风险

BWiki 图片可能包含游戏官方素材、社区整理截图和页面装饰资源。HSRChat 应将其定位为本地参考资料与引用来源，不应暗示拥有图片版权。

文档、README 或后续使用说明中应明确：

* 图片来源于 BWiki 页面引用或 BWiki 文件资源。
* 用途限定为剧情设定检索、考据和 Agent 上下文参考。
* 回答中引用图片时应保留来源页面信息。

### 10.3 质量与准确性风险

BWiki 是社区维护数据源，图片可能存在：

* 文件名不规范。
* 同图多名或同名替换。
* 图片与页面上下文错位。
* 上传者添加的非官方标注或拼接。
* 过期版本截图。

因此图片信源应继承现有 “Wiki 数据批判原则”，不得把图片存在本身视为绝对官方事实。

### 10.4 防爬与稳定性风险

BWiki 当前文本同步已经需要处理 WAF/EdgeOne 拦截。图片下载体积更大、请求更多，更容易触发限流。

后续实现应遵守：

* 低并发、强缓存、可恢复。
* 默认测试模式只解析少量页面。
* 支持只生成索引不下载图片。
* 明确 User-Agent、Referer 与重试退避。
* 不做盲目定时全量抓取。

### 10.5 多模态模型成本风险

如果为每张图生成 OCR、caption 或 embedding，会引入模型成本和缓存一致性问题。建议将“下载图片”和“生成派生语义”解耦：

* 第一阶段只建立图片资产索引。
* 第二阶段按需对少量高价值图片生成 OCR/caption。
* 第三阶段再考虑 embedding 与视觉相似检索。

## 11. 分阶段建议

### 阶段 0：需求确认（已完成）

目标：确认图片信源的范围和优先级。

建议决策点：

* 是否只做“剧情图片”，还是包含角色视觉资料。
* 是否允许图片二进制进入 Git。
* 是否需要 OCR/caption。
* 是否把图标类资源永久排除。

### 阶段 1：索引优先 MVP（已完成）

目标：只解析现有 `references/wiki/` 文本中的图片引用，生成 `references/bwiki_images/index.json`，暂不下载图片。

产出：

* 图片引用列表。
* 所属页面、分类、行号、上下文。
* 图片类型初步分类。
* 排除项与排除原因。

价值：

* 快速确认图片规模和质量。
* 不增加仓库二进制体积。
* 能为后续下载策略提供准确预算。

### 阶段 2：高价值图片下载（已完成）

目标：下载经过筛选的叙事图片。

优先级：

1. `{{图片放大|...}}` 中的任务 CG、物品照片、剧情截图。
2. `[[File:...]]` / `[[文件:...]]` 中位于书籍正文或任务正文的大图。
3. `{{角色对话|...|图片|...}}` 的短信图片。

暂不下载：

* 小尺寸图标。
* 奖励/材料/属性图。
* 页面装饰图。

阶段结果：

* 识别引用总数：265。
* 高价值去重候选：224。
* 已知大小并成功下载：208。
* 未解析候选：16，主要是少量失效文件名或远端不可解析文件。
* 实际下载体积：591,360,670 bytes，约 563.97 MiB。

## 12. 验收标准建议

后续若进入实现阶段，可用以下标准验收：

* 能从现有 Wikitext 中稳定识别主要图片语法。

* 能在测试模式下限制处理范围。
* 能在网络失败时保留已有索引，不写入空文件或损坏图片。
* 能通过 `git diff` 清晰审计新增/更新/删除的图片索引。
* 不影响现有 `sync_wiki.py` 和 `sync_bilibili.py` 的同步行为。

## 13. 已定规则与待确认问题

已定规则：

* 普通 Git 提交脚本、文档、`index.json`、`estimate_report.json`、`compressed_index.json` 和 `references/bwiki_images/assets_webp/`。
* 不提交 `references/bwiki_images/assets/` 原图缓存。图片二进制只提交 WebP，不提交原图缓存。
* 默认总量阈值为 1 GiB，超过即不下载。
* 默认排除小图标、机制图标、装饰图和无上下文 CSS 资源。
* 角色立绘属于有效高价值图片，但必须通过 `imageinfo` 解析原图，不保存缩略图。

在正式实现前，建议先确认以下产品与运维问题：

1. 是否需要对图片做 OCR，尤其是书籍插图、通缉令、卡片、短信截图？
2. 是否需要生成 caption、人物/地点/物件标签或视觉 embedding？
3. 是否需要把图片索引接入正式检索链路，而不是仅作为本地参考资产？
4. 后续若图片体积继续增长，是否引入 Git LFS 或外部对象存储？
5. 是否需要把图片同步配置写入 `config.json`，还是继续保持脚本显式调用？

## 14. 初步结论

“BWiki 附带图片作为多模态信源”是合理且有价值的增强，但它不应被理解为简单的“下载所有图片”。对 HSRChat 来说，真正需要的是一个可追溯、可筛选、可审计的视觉证据层：

* 以 BWiki 文本页面为父级来源。
* 以剧情、世界观、角色设定图片为核心范围。
* 默认排除战斗机制、奖励、材料、UI 小图标。
* 先建立索引，再按价值下载，再按需生成 OCR/caption/embedding。
* 保持与现有文本、视频同步流程解耦，避免影响已有 BWiki 文本和 B站元数据同步行为。

当前阶段已经完成索引、估算与高价值图片本地下载，并确认体积低于 1 GiB。后续重点不再是继续扩大抓取面，而是将 `index.json` 作为可审计入口接入检索流程，并按需为少量图片生成 OCR/caption 等派生语义。
