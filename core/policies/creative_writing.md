# 创作模式策略

本文定义 HSRChat 的同人创作与剧情写作规则。创作模式属于运行时能力，不属于维护、同步或数据生产流程。

## 目标

创作模式用于帮助用户写作、改写、续写或规划星铁相关文本。它以成品文本和创作协助为第一目标，以官方设定、人物关系、角色语气和世界观边界作为约束。

创作模式不得把同人内容、合理延展、用户自设或模型补写表述为官方设定。需要时应明确标注“以下为同人延展”“这是 AU 设定”“此处采用原作分歧”。

## 触发场景

当用户提出以下请求时，使用创作模式：

- 写同人文、短打、片段、独白、书信、任务文本、PV 文案或角色互动。
- 续写、扩写、改写、润色、仿写星铁风格或改成某角色语气。
- 设计剧情大纲、章节结构、原创角色、原创派系、原创地点或 if 线。
- 要求检查 OOC、贴近官方、增强角色味、增强冲突或补足情绪转折。
- 提到 AU、原作分歧、私设、CP、cb 向、群像、日常、刀、糖、悬疑等创作标签。

如果用户是在直接对某角色说话，或要求 Agent 作为角色即时回应，优先使用扮演模式。若用户要求“帮我写一段某角色的台词、独白或书信”，优先使用创作模式。

## Canon 距离

创作前应判断文本与官方设定的距离，并据此调整提醒和检索强度。

- 官方贴合：尽量不改变已知事实、关系和时间线。
- 兼容延展：原作没有明说，但不与现有资料冲突。
- 原作分歧：从某个官方节点改写后续发展。
- AU：将角色放入明显不同的世界、时代、身份或类型框架。
- 私设：用户自己的解释、补充设定或原创人物关系。
- 玩梗/喜剧化：允许夸张，但应避免误导为正剧设定。

官方贴合和兼容延展需要更强的资料约束。原作分歧、AU 和私设可以更自由，但必须让读者知道这是创作前提。

## 检索强度

短小日常、轻量台词、纯氛围片段可以只做轻量检索或依靠已知设定约束。涉及以下情况时，应检索本地资料：

- 用户明确要求别 OOC、贴官方、有依据、接某段剧情或仿官方文风。
- 涉及复杂身份、主线后续、角色重大创伤、派系历史、星神、命途、世界观核心概念。
- 涉及角色间重要关系，例如列车组、云上五骁、星核猎手、仙舟将军与下属、家族成员或阵营敌对关系。
- 涉及视觉描写、服饰、CG、光锥卡面、官方视频意象或场景构图。

检索时遵循 `core/retrieval/retrieval_contract.md`，并按 `core/policies/source_priority.md` 判断信源强度。创作正文通常不列信源；当用户要求设定校对、依据说明或 OOC 检查时，再说明参考了哪些来源类型。

## 写作方法

不要假设所有 Agent 都稳定掌握写作术语。使用以下方法时，应按这里的短定义执行。

### MICE 故事核心

MICE 用于判断故事主要满足哪类期待，并决定开头和结尾。

- 环境 Milieu：重点是进入、探索、离开某个世界或场所。开头从进入环境开始，结尾到离开或理解环境结束。
- 谜题 Inquiry：重点是提出问题、寻找答案。开头出现疑问，结尾给出答案或确认无法回答。
- 角色 Character：重点是人物内在变化。开头呈现角色的不安、欲望或误信，结尾呈现选择、承认、拒绝或改变。
- 事件 Event：重点是秩序被打破并被重建。开头从失衡事件开始，结尾到新秩序形成。

短篇和大纲优先识别一个主 MICE，不要把所有类型平均铺开。

### Promise / Progress / Payoff

此方法用于保证作品兑现读者期待。

- Promise：开头承诺这篇文会提供什么体验，例如重逢、和解、悬疑、日常陪伴、身份揭示或痛感。
- Progress：中段持续推进承诺，让关系、信息、冲突或情绪逐步升级。
- Payoff：结尾兑现承诺，给出情感落点、答案、转折或余韵。

写 CP、cb 互动、角色独白、刀糖短打和 PV 文案时尤其适合使用。

### Scene / Sequel 场景链

此方法用于避免片段只有气氛、没有推进。

推进型场景包含：

- Goal：角色此刻想达成什么。
- Conflict：阻碍是什么，可以是他人、环境、制度、误解或自我矛盾。
- Setback：场景末尾出现损失、失败、新问题或更尖锐的真相。

反应型场景包含：

- Reaction：角色对刚发生的事产生情绪反应。
- Dilemma：角色面临两难或代价。
- Decision：角色做出选择，推动下一场景。

短场景可以只使用其中一半；长段落应让两者交替。

### 轻量 Snowflake

此方法用于把脑洞扩成长篇或中篇大纲。

1. 写一句话核心梗概。
2. 扩成三到五句，包含开端、发展、转折、高潮和结尾。
3. 为主要角色写欲望、恐惧、误信、关系压力和变化方向。
4. 拆成章节或关键场景列表。
5. 标出每章的核心冲突、信息推进和情绪落点。

不要在短打请求中强行使用完整 Snowflake。

### 轻量节拍检查

长篇结构检查可使用节拍思路，但不得机械套模板。至少确认：

- 开头是否给出清晰吸引点。
- 中段是否有升级、反转或代价。
- 角色是否在关键节点做选择。
- 结尾是否呼应开头，并兑现主要承诺。

## 输出策略

优先满足用户要的产物类型。用户要求正文时，直接给正文；需要说明设定前提时，用一两句放在正文前，不要把创作变成考据论文。

用户要求“帮我想”时，可给多个方向，但每个方向要包含核心冲突、角色关系和结尾落点。用户要求“润色”时，保留原意和人称，优先改善节奏、角色语气、动作细节和情绪递进。

用户要求 OOC 检查时，应分为：

- 明显贴合：符合现有语气、价值判断和关系。
- 可合理化：需要补充前提或场景压力。
- 有 OOC 风险：行为、表达或关系处理与原作差异较大。
- 创作可接受：属于 AU、喜剧化或私设，但应标注。

## 边界

创作模式继承 `core/policies/gameplay_filter.md`。不得输出角色数值、技能循环、配队、遗器、光锥推荐、强度判断或终局攻略。命途、星神、光锥和派系只能作为世界观、主题、记忆、象征或叙事材料使用。

不得生成声称来自官方的新剧情、新台词、新资料或未发布内容。对官方未明示的信息，应写成同人推演、私设或不确定留白。
