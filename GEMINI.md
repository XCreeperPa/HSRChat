# HSRChat 项目开发指南

本项目是一个专用于《崩坏：星穹铁道》世界观与剧情交流的 **Agent Skill (HSRChat)** 仓库。

## 核心设计与规约

### 1. 专注范围
本技能仅处理与《崩坏：星穹铁道》世界观、历史事件、剧情分支、角色背景设定相关的内容。

### 2. 机制与数值的过滤规约
* **现状**：从 BWiki 爬取回来的原始语料（存储在 `references/wiki/` 下）在“角色”、“冒险任务”等分类中可能夹杂少量的游戏机制描述、晋阶材料、技能定位、副本解锁流程等与剧情无关的数值/战斗信息。
* **处理策略**：**不需要编写物理清洗脚本**。我们通过在 Skill 指令（`SKILL.md`）中明确添加约束，使 Agent 在被激活并读取 `references/wiki/` 数据时，能够**自发且动态地过滤掉**其中的战斗、数值、机制、光锥遗器配队等非剧情世界观内容。

## 数据安全与版本控制规约 (Git-based Version Control)

为了防御网络爬取时遭遇恶意篡改、Wiki 页面内容被意外清空或防护拦截（如 TEO 567）产生坏数据覆盖本地完整数据，项目遵循以下版本控制流程：

### 1. 单仓库控制策略 (Single Git Repository)
项目采用**方案二：仅在项目根目录进行单 Git 仓库管理**。整个项目（包括脚本、分类配置 `config.json`、历史同步状态 `state.json` 以及 `references/wiki/` 中的纯文本数据）全部纳入同一个根目录 Git 仓库管理中。
* 数据修改与配置变更的版本保持强关联，禁止在 `references/wiki/` 内部初始化独立的子仓库或使用 Git Submodule。

### 2. 同步与回滚工作流 (Sync & Rollback Workflow)

在运行爬虫更新数据时，必须执行以下三步安全工作流：

#### 第一步：同步前状态检查
在运行 `scripts/crawler.py` 之前，确保当前工作区是干净的。运行：
```bash
git status
```
如有未提交的代码或数据更改，建议先执行 `git commit` 或 `git stash` 保护工作区。

#### 第二步：运行同步
运行爬虫脚本，等待全量同步完成。
```bash
python scripts/crawler.py
```

#### 第三步：数据审核与提交/回滚
同步完成后，**禁止立即提交所有更改**。必须先对数据进行合规性审计：
* **预览修改**：使用以下命令查看哪些文件被修改或删除了：
  ```bash
  git status
  ```
* **审计差异**：对修改过的文本，特别是由大体积变小体积的文件，执行 diff 检查内容是否被恶意清空或篡改为无意义 HTML 拦截块：
  ```bash
  git diff references/wiki/
  ```
* **提交归档（数据安全无误）**：如果确认下载回来的全部是正常且完整的剧情文本，执行提交归档：
  ```bash
  git add .
  git commit -m "data: 同步星穹铁道 Wiki 最新剧情与设定数据"
  ```
* **一键回滚（存在坏数据/误删）**：如果发现某些页面在同步后变成了空文件、被删除或者内容被篡改，**直接放弃本次同步结果，一键安全回滚**：
  ```bash
  git restore references/wiki/
  git restore references/wiki/state.json
  ```
  *(注：必须同时回滚 state.json，以使状态与本地物理数据完全对齐，确保下次运行能够重新请求该页面)*

## 目录结构说明
* `SKILL.md`：Skill 核心声明与触发指令。
* `config.json`：定义要爬取的目标世界观分类。
* `references/wiki/`：保存抓取的 Wiki 文本语料（按分类分文件夹存储）。
* `references/wiki/state.json`：存储已下载条目版本时间戳的本地同步状态文件。
* `scripts/`：包含全并发爬虫脚本。
